Design Doc · v1 spec

OnTrack
Know what you built.

You build a project with an AI agent — it picks the libraries, writes the auth, wires the config. You ship a working app with no idea what's inside it, or whether you could have built it yourself. OnTrack reads the project, works out what the AI leaned on, asks you a few questions to gauge what you know, and hands back a step-by-step path of what to learn.

A route, not a course Asks — doesn't assume Does not teach Claude Code plugin

Everything sits on the .ontrack plain-JSON format. Portability is a property of the format — not of an abstraction layer, because there isn't one.

Changed from the first cut: the old design refused to guess what you knew — it just listed what the project used and let you self-mark each item with a radio. This version asks instead of assumes: a short assessment replaces the radios, and the output is an ordered path instead of a flat inventory. Passive capture underneath is unchanged.

Resuming · session cursor

Pick up from the last unprocessed session

The hook drops a session marker line at the end of every session, so evidence.jsonl knows where one session's changes stop. state.json holds one value — the last session already interpreted. /ontrack reads evidence after that cursor, does the expensive work only on what's new, then advances it. No second log to keep in sync; the append-only evidence file is the session register.

incremental · LLM

Concept inference + question authoring run only over files touched in new sessions. This is the cost the cursor saves.

gated by state.json

always full · deterministic

Staleness validation re-checks the whole set against the current repo — a dep removed two sessions ago only drops out on a full pass. No LLM, so full is free.

never gated

Delete state.json → full reprocess (fresh-clone behaviour). The cursor makes re-runs cheap without letting removed things linger.

The assessment

Two question modes — graded where checkable, self-report where not

Hybrid by design. The LLM picks a mode per concept: concrete, checkable facts become multiple-choice with a correct answer; judgment concepts (system design, security posture), where "correct" is situational, become self-report.

graded

MCQ with a clean correct answer (an answer index). The server grades by comparing the choice to the key.

No LLM at grade time — cheap, offline, no fuzzy judgment.

used for: language features, framework APIs, tooling

self_report

No correct answer — the user rates their own comfort (Confident / Shaky / New) with an optional free-text note.

Never graded. Keeps the honesty line where truth is situational.

used for: system design, security posture
questions.json — one graded, one self_report
{ "questions": [
  { "id": "q:react/useeffect", "concept": "concept:react/useeffect",

domain ∈ language · framework · system-design · security · tooling — the path buckets. level orders within a bucket. Every concept must be a real inventory id; orphaned questions drop on rebuild.

Intake + scoring

Answer → concept state → path

Intake first: the user declares an overall level once — beginner / intermediate / senior. Level scales path depth, not which questions get asked: everyone answers, but a senior's basics they get right drop straight to Learned instead of burying the path. A senior can still have gaps, so the answer decides — nothing is pre-skipped.

Components · v1

Three parts

01

Tracker — Claude Code SessionEnd hook

Fires once per session. Writes observations only from concrete signals: manifests (package.json, requirements.txt, Cargo.toml, go.mod), file extensions, config files. One line per fact. No dedup, no interpretation, no state read.

02

Skill /ontrack — regenerate + author questions + serve

Read evidence + the state.json cursor; validate the whole deduped set vs. repo → confirmed items (staleness pass stays full). LLM infers concepts only for sessions after the cursor (React → useEffect, JSX, props). Author a question for every concept with no question yet, or whose code changed since its answer — pick domain, pick mode, write options (+ answer for graded). Incremental: never re-ask an answered, unchanged concept. Write inventory + questions, advance the cursor, start the server.

It regenerates data + questions only — does not generate HTML, does not grade, never asks in chat. The dashboard owns intake, asking, and the path.

03

Dashboard — local server + committed dashboard.html

Stdlib only, local-only, security real:

  • Python http.server / one-file Node. No deps.
  • Bind 127.0.0.1, port 3874 — never 0.0.0.0. File-writing endpoint stays off the network. Port in use → increment and reprint.
  • POST /answer → validate vs. questions.json, grade MCQ (choice == answer), atomic write. POST /level → validate vs. 3 values, atomic write.
  • personal.json is the only file it writes; everything else read-only. Dies on idle / Ctrl-C.

The dashboard asks what you know and shows what to learn + where to search. It does NOT teach the material.

Build order

Format first, path last

  1. Data format.

    evidence (+ session marker) + inventory + questions + personal + state.json cursor. Format before anything.

  2. Tracker hook.

    Writes confirmed evidence + a session marker per run. Prove passive capture.

  3. Build inventory.

    /ontrack builds inventory from confirmed evidence. Prove the core loop.

  4. LLM inference pass.done

    inferred / possible concepts with a parent.

  5. Question authoring pass.

    questions.json — domain, mode, options, answer. One per concept, incremental.

  6. Dashboard + server.

    Intake level, Questions section, POST /answer + /level, server-side MCQ grading.

  7. Learning Path section.

    Derive state per concept, group by domain, order by level, live re-sort.

  8. Post-v1.

    Re-verify learning when code changes; author-signal weighting.

The honesty line

Confidence is about detection, not knowledge

Two separate axes. Confidence (on inventory items, never on evidence) is how sure OnTrack is that the project uses a thing. Concept state (Learned / Review / Learn) is what the user knows, and comes only from their answers.

ConfidenceHow /ontrack derives itShown
confirmedDeterministic observation validated vs. repoyes
inferredLLM pass, from code with file:line evidenceyes
possibleLLM pass, weak signalhidden by default

The hook only writes observations — no confidence field. The server only grades MCQs — no LLM. Confidence is assigned at interpretation time; knowledge comes from the user. Neither guesses the other.

Verification

How we know it holds

  • Tracker: throwaway React+auth project → evidence.jsonl auto-fills (react, jsonwebtoken, tsx) with zero manual input.
  • Questions: /ontrack writes a graded MCQ for useEffect (with answer) and a self_report probe for JWT storage. Every concept is a real inventory id.
  • Intake: no level → picker first; pick beginner → personal.json gets "level":"beginner", questions render.
  • Grading: answer useEffect correctly → concept Learned, out of path; wrong → Learn, appears in Framework bucket. self=Shaky → Review, in path.
  • Path: groups by domain, numbers within group, orders by level, re-sorts live on each answer, no reload.
  • Isolation: server only writes personal.json; inventory + questions byte-identical after answering.
  • Incremental: re-run /ontrack with no code change → answered questions kept, not re-asked; change a concept's code → its question re-authored.
  • Cursor: after a run state.json equals the newest session marker; a new session → only its files hit the LLM pass; delete state.json → full reprocess; a dep removed before the cursor still drops out (validation stayed full).
  • Honesty: no answer or question ever lands in evidence.jsonl; the server never runs the LLM to grade.
