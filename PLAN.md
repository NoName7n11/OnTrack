# OnTrack — "Know What You Built"

## Context / Problem
People build projects with AI agents (Claude Code, Codex) but **don't know what
they built**. The agent picks the libraries, writes the auth, wires the config —
user ships a working app with no idea what's inside it, or whether they could
have built it themselves.

OnTrack bridges that gap: it turns the project into a **structured learning
path**. On demand, it traverses the project, figures out the topics/concepts the
AI leaned on (language features, framework APIs, system design, application
security, tooling), and **asks the user a few questions** to gauge what they
already know. From the answers it produces a step-by-step path of what to learn,
grouped by domain.

Goal is **a route, not a course**. OnTrack does not teach the material — it names
what the project is built on, checks what the user knows, and hands an ordered
list of what to go learn (with a search term per item). Learning happens outside
the tool.

Repo is packaged as a Claude Code plugin marketplace for alpha testing.

## What changed from the first cut (self-classify → assess)
The first design refused to guess what the user knew — it only listed what the
project used and let the user self-mark each item with a radio. This version
**asks instead of assumes**: a short assessment (graded multiple-choice where a
clean answer exists, self-report where it doesn't) replaces the radios, and the
output is an ordered learning path rather than a flat inventory. The passive
capture layer (evidence → inventory) is unchanged; the interpretation and
dashboard on top of it are new.

## Architecture — data format is the product
The `.ontrack` data format is the core. Everything else is an adapter or interface
around it. Design the format tool-agnostic; **do not build an abstraction layer** —
portability is a property of the plain-JSON format, not of extra code.

- **Core**: `.ontrack/` data format (evidence + inventory + questions + personal).
- **Adapter 1** (v1): Claude Code `SessionEnd` hook — appends evidence.
- **Interface 1** (v1): Claude skill `/ontrack` — builds inventory, authors
  questions, serves the dashboard.
- **Future**: Codex wrapper/command, Cursor rule, standalone CLI. Each is "another
  script that appends to `evidence.jsonl`" — cheap to add *because* no adapter
  layer exists.

## Data model — four lifecycles, never mixed
Four kinds of data, each with a different owner and lifecycle.

**Evidence** = facts *observed at a point in time*. Append-only, hook-written.
NOT eternal truth — a dep may be removed later, a `.tsx` file deleted. Evidence
records what was seen when; it can go stale.
**Inventory** = current interpretation of *what the project uses*, regenerated
from evidence **+ current repo state**. Stale evidence drops out.
**Questions** = the assessment: one probe per concept, LLM-authored, tying a
concept to a domain and a question the dashboard can render.
**Personal** = the user's intake level + answers. The only file the dashboard
writes. Drives the path; never touches the other three.

### `.ontrack/evidence.jsonl` — append-only observation log (committed)
Hook appends one line per observed fact. No read-modify-write, no dedup, no merge
pain. Duplicates allowed and meaningful (same fact at T2 = "still present then").
Keys are **structured** — don't collapse to vague `type=file, name=.tsx` (every
file collides). Split extension vs command args:
```json
{"type":"dependency","name":"react","source":"package.json","detected_at":"2026-07-11T10:00:00Z"}
{"type":"command","name":"npm install","args":["jsonwebtoken"],"source":"bash","detected_at":"2026-07-11T10:02:00Z"}
{"type":"file_extension","name":"tsx","source":"src/App.tsx","detected_at":"2026-07-11T10:03:00Z"}
```

### `.ontrack/inventory.json` — derived, regenerated by `/ontrack` (committed)
Built from evidence validated against current repo. Each item has a **stable
`id`**, derived deterministically from `kind` + `parent` + `name` (slugified):
`library:react`, `concept:react/useeffect`, `tool:npm`. `parent` (optional, an
`id`) nests concepts under their library. Confidence is assigned here at
interpretation time — never on evidence.
```json
{
  "items": [
    { "id": "library:react", "name": "React", "kind": "library",
      "confidence": "confirmed", "what": "Component-based UI library",
      "where": ["package.json", "src/App.tsx"], "search": "react beginner tutorial" },
    { "id": "concept:react/useeffect", "name": "useEffect", "kind": "concept",
      "parent": "library:react", "confidence": "inferred",
      "what": "Run side effects after render", "where": ["src/App.tsx:14"],
      "search": "react useeffect tutorial" }
  ]
}
```
**Anti-churn**: stable key ordering; write only when content differs; no per-run
`generated_at` in the committed body.

### `.ontrack/concepts.json` — LLM-authored concept layer (committed)
Written by the `/ontrack` inference pass (Claude), kept **separate from
inventory.json** so the deterministic `build.py` never clobbers it. Holds only
`inferred`/`possible` concept items, each with a `parent` that must be a confirmed
inventory id. `build.py` validates and merges these into `inventory.json`:
orphaned concepts (parent gone) are dropped; `where` file paths that no longer
exist are pruned; `confirmed` always wins on an id collision.

### `.ontrack/questions.json` — the assessment (committed)
LLM-authored, one question per concept worth asking. Ties a concept to a
`domain` and a question the dashboard renders. Two modes:
- **`graded`** — multiple-choice with a clean correct answer. Carries an `answer`
  index. Used for concrete, checkable facts. The **server** grades by comparing
  the chosen index to `answer` — no LLM at grade time (cheap, offline, no fuzzy
  judgment).
- **`self_report`** — no correct answer; the user rates their own comfort. Used
  for judgment concepts (system design, security posture) where "correct" is
  situational. Never graded.

`domain` ∈ `language | framework | system-design | security | tooling` — this is
what the learning path buckets by. `level` ∈ `basic | intermediate | advanced`
orders questions within a domain.
```json
{ "questions": [
  { "id": "q:react/useeffect", "concept": "concept:react/useeffect",
    "domain": "framework", "mode": "graded", "level": "basic",
    "prompt": "What does useEffect do?",
    "options": ["Runs side effects after render", "Styles a component", "Defines a route"],
    "answer": 0 },
  { "id": "q:auth/jwt-storage", "concept": "concept:auth/jwt",
    "domain": "security", "mode": "self_report", "level": "intermediate",
    "prompt": "How comfortable are you with where JWTs are stored and why?",
    "options": ["Confident", "Shaky", "New"] }
] }
```
Each question's `concept` must be a real inventory id. Orphaned questions (concept
gone) are dropped on rebuild, same rule as concepts.

### `.ontrack/personal.json` — private per-user state (gitignored)
**The only file the dashboard writes.** Holds the intake `level` and one answer
per question, keyed by question `id`. Not inferred — the user tells it. Affects
**display + path only**, never touches the other files.
```json
{ "level": "beginner",
  "answers": {
    "q:react/useeffect": { "choice": 0, "at": "2026-07-20T10:00:00Z" },
    "q:auth/jwt-storage": { "self": "Shaky", "note": "not sure about refresh", "at": "2026-07-20T10:01:00Z" }
  } }
```
`level` ∈ `beginner | intermediate | senior` (set once at intake, editable).
`graded` answers store `choice` (index); `self_report` answers store `self` (the
chosen label) + optional free-text `note`.

## Intake — declare level once
Before questions, the dashboard asks the user's overall level. **Level scales path
depth, not which questions are asked** — everyone answers, but for a `senior` the
`basic`-level concepts they get right drop straight to Learned and never clutter
the path. A senior can still have gaps, so we don't pre-skip; we let the answer
decide and just avoid burying them in fundamentals they clearly know.

## Answer → concept state (deterministic)
The server never interprets; it grades MCQs and stores answers. The **dashboard**
derives each concept's state from its answer for display + path placement:

| answer                                   | state    | in path? |
|------------------------------------------|----------|----------|
| graded correct · self=`Confident`        | Learned  | no       |
| self=`Shaky`                             | Review   | yes      |
| graded wrong · self=`New`                | Learn    | yes      |
| *(unanswered)*                           | To Ask   | no (asked first) |

## Confidence levels — enforce the honesty line
Separate from knowledge state above; this is about *detection* certainty, on
inventory items only, never on evidence.

| level       | how `/ontrack` derives it                        | shown |
|-------------|--------------------------------------------------|-------|
| `confirmed` | deterministic observation validated vs repo      | yes   |
| `inferred`  | LLM pass, from code with file:line evidence       | yes   |
| `possible`  | LLM pass, weak signal                             | hidden by default |

**Hard rule: the hook only writes observations — no confidence field.** An
observation becomes a `confirmed` inventory item only after `/ontrack` validates
it against the repo. All `inferred` / `possible` items come from the on-demand LLM
pass. Boring facts in, confidence assigned at interpretation time.

## Components (v1)

### 1. Tracker — Claude Code `SessionEnd` hook
Fires once per session (cheap, quiet). Writes **observations only** (no confidence
field), from concrete signals:
- Manifests: `package.json`, `requirements.txt`, `Cargo.toml`, `go.mod`, etc.
- File extensions + config files (`.tsx`, YAML, Dockerfile, CI configs).
- Command-run observations are deliberately deferred until hook payload support is
  clear.

Appends each as one line to `evidence.jsonl`. Dumb by design — no dedup, no
interpretation, no state read.

### 2. Skill `/ontrack` — regenerate + author questions + serve
On demand, MVP flow:
1. Read `evidence.jsonl`, dedup by structured key, **validate each against current
   repo** (dep still in manifest? file still exists?). Stale drops out. Build
   `confirmed` items.
2. Run a **light LLM inference pass** over touched files → `inferred` / `possible`
   concept items with `parent` + file:line evidence. (Where "React" becomes
   "useEffect", "JSX", "props" — the granularity that matters.)
3. **Author questions** for every concept that has no question yet, or whose code
   changed since the last answer. Pick `domain`, pick `mode` (graded vs
   self_report), write options (+ `answer` for graded). **Incremental** — never
   re-ask an answered, unchanged concept. Reuses the re-verify check
   (`answer.at` vs `where` last-modified). Write `questions.json`.
4. Write `inventory.json` = everything (stable order, only if changed).
5. Start local server, print: `Open http://localhost:3874`.

`/ontrack` regenerates data + questions only — it does NOT generate HTML, does NOT
grade, and never asks the questions in chat. The dashboard owns intake, asking,
and the path.

### 3. Dashboard — local server + `dashboard.html` (committed template)
`dashboard.html` is a **static, data-driven template committed to the repo** (it's
source code, not a generated artifact — churns only when the UI improves). At
runtime it `fetch`es `inventory.json` + `questions.json` (project truth) +
`personal.json` (user state) and renders client-side.

Two sections:
- **Questions** — an intake level picker first (if `level` unset), then a card per
  unanswered question: prompt · radio options · optional text note (`self_report`
  only). Submit → `POST /answer`.
- **Learning Path** — derived, grouped by `domain` (Language / Framework / System
  design / Security / Tooling), numbered within each group, ordered by concept
  `level`. Shows only Learn/Review concepts. Each step: concept · what · used-in
  (file:line) · search query. Re-sorts live as questions get answered.

Server scope (local-only + security):
- **Stdlib only** (Python `http.server` / Node one-file). No deps. Serves the
  static `dashboard.html` + the three JSON files (GET) and one POST. JSON
  validation + port fallback + atomic write + path safety is real work — small but
  not trivial.
- **Bind `127.0.0.1`, port `3874`.** Never `0.0.0.0` — the file-writing endpoint
  stays off the network. Port in use → increment and reprint.
- **Two write paths, one file: `POST /answer`** → `{question_id, choice|self, note?}`
  → validates against `questions.json` (id exists, choice in range / self in the
  option set), grades MCQs (`choice == answer`), writes `personal.json`
  **atomically** (temp + rename). Also `POST /level` → `{level}` → validates
  against the 3 allowed values → same atomic write. `personal.json` is the only
  file it ever writes; everything else is read-only here.
- Live DOM re-sorts on submit; no page reload, no `/ontrack` re-run to save answers.
- Lifecycle: dies on idle timeout / Ctrl-C.

**Boundary held:** the dashboard asks what you know and shows what to learn +
where to search. It does NOT teach the material.

The loop: build → `/ontrack` (capture + author questions) → open dashboard →
declare level → answer questions → path fills + re-sorts → go learn the path
elsewhere.

## Post-v1
- **Re-verify learning** *(evidence-driven)*: `answer.at` already timestamps each
  answer; on rebuild, if a `Learned` concept's `where` code changed after that,
  re-ask its question. Tests the evidence, not the user's memory — keeps the
  honesty boundary. No memory quizzes.
- **Author-signal weighting**: down-rank concepts the user hand-wrote (they
  probably know them), up-rank Claude-wrote-untouched code. Weak proxy — layer on
  after the core assessment proves useful.

## Deferred (still not built)
- LLM-graded free-text answers (fuzzy, expensive — MCQ + self-report covers v1).
- Cross-concept prerequisite graph (domain buckets + `level` ordering is enough).

## Packaging
Shipped as a Claude Code **plugin** in the `ontrack/` subdir; this repo is also the
**marketplace** (`.claude-plugin/marketplace.json`). Install:
`/plugin marketplace add NoName7n11/OnTrack` → `/plugin install ontrack@ontrack`.
Plugin-internal command paths use `${CLAUDE_PLUGIN_ROOT}`; data still writes to the
**user project's** `.ontrack/` (hooks/skills run with the project as cwd).

## Build order
1. Data format: `evidence.jsonl` + `inventory.json` + `questions.json` + `personal.json` schemas. Format first.
2. Tracker hook → writes confirmed evidence. Prove passive capture.
3. `/ontrack` builds `inventory.json` from confirmed evidence. Prove core loop.
4. LLM inference pass → `inferred`/`possible` concepts + `parent`. **[done]**
5. Question authoring pass → `questions.json` (domain, mode, options, answer).
6. Dashboard + server: intake level, Questions section, `POST /answer` + `POST /level` → `personal.json`, server-side MCQ grading.
7. Learning Path section: derive state per concept, group by domain, order by level, live re-sort.
8. (Post-v1) re-verify learning; author-signal weighting.

## Verification
- Tracker: build a throwaway React+auth project in a session → confirm
  `evidence.jsonl` auto-fills (react, jsonwebtoken, tsx) with zero manual input.
- Inventory: run `/ontrack` → `inventory.json` lists `confirmed` libs +
  `inferred` concepts (with `id` + `parent`), each with what/where/search.
- Questions: `/ontrack` writes `questions.json` — a `graded` MCQ for `useEffect`
  (with `answer`), a `self_report` probe for JWT storage. Every `concept` is a
  real inventory id; no question for a concept that doesn't exist.
- Intake: open `localhost:3874` with no `level` → level picker shows first; pick
  `beginner` → `personal.json` gets `"level":"beginner"`; questions then render.
- Grading: answer the `useEffect` MCQ correctly → server records the answer, the
  concept shows Learned and stays out of the path; answer wrong → shows Learn and
  appears in the Framework bucket. `self_report`=`Shaky` → Review, in path.
- Path: Learning Path groups by domain, numbers within group, orders by concept
  `level`, and re-sorts live on each answer with no reload.
- Isolation: confirm the server only ever writes `personal.json`; `inventory.json`
  and `questions.json` byte-identical after answering.
- Incremental: answer a question, re-run `/ontrack` with no code change → that
  question keeps its answer and is not re-asked; change the concept's code →
  its question is re-authored / re-asked.
- Honesty: no answer or question ever appears in `evidence.jsonl` (hook writes
  facts only); the server never runs the LLM to grade.
