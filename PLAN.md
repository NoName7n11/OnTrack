# OnTrack — "Know What You Built"

## Context / Problem
People build projects with AI agents (Claude Code, Codex) but **don't know what
they built**. The agent picks the libraries, writes the auth, wires the config —
user ships a working app with no idea what's inside it.

OnTrack bridges that gap. Goal is **awareness, not curriculum**. The AI passively
tracks what the project uses as it grows. Any time the user wants, they open an
inventory: "here's everything your project uses." They scan it, spot anything
they don't recognize, and go learn it themselves (YouTube, docs, etc).

Key design consequence: **the tool does not teach**, and **it does not claim to
know what the user knows**. It surfaces what the project uses + explains each item
in one line + hands a good search term. The user decides what they know; learning
happens outside the tool. This keeps it small and honest.

Repo is packaged as a Claude Code plugin marketplace for alpha testing.

## Architecture — data format is the product
The `.ontrack` data format is the core. Everything else is an adapter or interface
around it. Design the format tool-agnostic; **do not build an abstraction layer** —
portability is a property of the plain-JSON format, not of extra code.

- **Core**: `.ontrack/` data format (evidence + inventory + personal).
- **Adapter 1** (v1): Claude Code `SessionEnd` hook — appends evidence.
- **Interface 1** (v1): Claude skill `/ontrack` — builds inventory, serves dashboard.
- **Adapter 2**: Codex local wrapper (`codex/ontrack.py`) — records observations,
  builds inventory, and serves the dashboard without Claude plugin install.
- **Future**: Cursor rule, standalone CLI. Each is "another script
  that appends to `evidence.jsonl`" — cheap to add *because* no adapter layer exists.

## Data model — evidence vs inventory (the key split)
Two different kinds of data with different lifecycles. Never mix them.

**Evidence** = facts *observed at a point in time*. Append-only, not overwritten.
NOT eternal truth — a dep may be removed later, a `.tsx` file deleted. Evidence
records what was seen when; it can go stale.
**Inventory** = current interpretation, regenerated from evidence **+ current repo
state**. `/ontrack` validates each evidence item against the repo now, so a removed
dep drops out of inventory even though its evidence line stays in the log.

### `.ontrack/evidence.jsonl` — append-only observation log (committed)
Hook appends one line per observed fact. No read-modify-write, no dedup, no merge
pain. Duplicates allowed and meaningful (same fact at T2 = "still present then").
Keys are **structured** — don't collapse to vague `type=file, name=.tsx` (every
file collides). Split extension vs command args:
```json
{"type":"dependency","name":"react","source":"package.json","detected_at":"2026-07-11T10:00:00Z"}
{"type":"command","name":"npm install","args":["jsonwebtoken"],"source":"bash","detected_at":"2026-07-11T10:02:00Z"}
{"type":"file_extension","name":"tsx","source":"src/App.tsx","detected_at":"2026-07-11T10:03:00Z"}
```

### `.ontrack/inventory.json` — derived, regenerated by `/ontrack` (committed)
Built from evidence validated against current repo. References evidence by
structured natural key `(type, name[, args])` — no generated IDs. Each item carries
what the user needs to self-classify + self-learn.
**Anti-churn**: stable key ordering; write only when content differs; no per-run
`generated_at` inside the committed body (avoids timestamp-only diffs). Contains
**everything** — `personal.json` never removes items here, only affects display.
Each item has a **stable `id`**, derived deterministically from `kind` + `parent` +
`name` (slugified): `library:react`, `concept:react/useeffect`, `tool:npm`. Status
is keyed by `id`, not display name — survives renames and same-name collisions.
```json
{
  "items": [
    {
      "id": "library:react",
      "name": "React",
      "kind": "library",
      "confidence": "confirmed",
      "what": "Component-based UI library",
      "where": ["package.json", "src/App.tsx"],
      "search": "react beginner tutorial",
      "from": [{"type":"dependency","name":"react"}]
    },
    {
      "id": "concept:react/useeffect",
      "name": "useEffect",
      "kind": "concept",
      "parent": "library:react",
      "confidence": "inferred",
      "what": "Run side effects after render",
      "where": ["src/App.tsx:14"],
      "search": "react useeffect tutorial",
      "from": [{"type":"dependency","name":"react"}]
    }
  ]
}
```
`parent` (optional, an `id`) nests concepts under their library for the dashboard.
Hierarchy guides *which* questions to ask — it NEVER assumes: React "known" does not
mark `useEffect` learned. Each node is answered independently.

### `.ontrack/concepts.json` — LLM-authored concept layer (committed)
Written by the `/ontrack` inference pass (Claude), kept **separate from
inventory.json** so the deterministic `build.py` never clobbers it. Holds only
`inferred`/`possible` concept items, each with a `parent` that must be a confirmed
inventory id. `build.py` validates and merges these into `inventory.json`:
orphaned concepts (parent gone) are dropped; `where` file paths that no longer
exist are pruned; `confirmed` always wins on an id collision.
```json
{ "concepts": [
  { "id": "concept:react/useeffect", "name": "useEffect", "parent": "library:react",
    "confidence": "inferred", "what": "Run side effects after render",
    "where": ["src/App.tsx:14"], "search": "react useeffect tutorial" }
] }
```

### `.ontrack/personal.json` — private per-user state (gitignored)
**The only file the dashboard writes.** Per-item learning status, keyed by `id`.
Not inferred — user *tells* it. Affects **display only**, never touches
`inventory.json`. Four states: `known` · `somewhat` · `to_learn` · `ignored`.
Unlisted = unanswered.
```json
{ "status": { "library:react": "somewhat", "concept:react/jsx": "known", "concept:react/useeffect": "to_learn", "tool:docker": "ignored" } }
```

Status → dashboard section (deterministic):

| status         | section     |
|----------------|-------------|
| *(unanswered)* | To Review   |
| `to_learn`     | Learning    |
| `somewhat`     | Learning    |
| `known`        | Learned     |
| `ignored`      | Ignored     |

## Confidence levels — enforce the honesty line
Fixes "concrete detection" vs guesses like "JWT refresh flow". **Confidence exists
only on inventory items**, never on evidence. `/ontrack` assigns it:

| level       | how `/ontrack` derives it                        | shown |
|-------------|--------------------------------------------------|-------|
| `confirmed` | deterministic observation validated vs repo      | yes   |
| `inferred`  | LLM pass, from code with file:line evidence       | yes   |
| `possible`  | LLM pass, weak signal                             | hidden by default |

**Hard rule: the hook only writes observations — no confidence field.** An
observation becomes a `confirmed` inventory item only after `/ontrack` validates it
against the repo. All `inferred` / `possible` items come from the on-demand LLM
pass. Boring facts in, confidence assigned at interpretation time.

## Components (v1)

### 1. Tracker — Claude Code `SessionEnd` hook
Fires once per session (cheap, quiet). Writes **observations only** (no confidence
field), from concrete signals:
- Manifests: `package.json`, `requirements.txt`, `Cargo.toml`, `go.mod`, etc.
- File extensions + config files (`.tsx`, YAML, Dockerfile, CI configs).
- Command-run observations are deliberately deferred until hook payload support is
  clear.

Appends each as one line to `evidence.jsonl`. Dumb by design — no dedup, no
interpretation, no state read.

### 2. Skill `/ontrack` — regenerate + serve dashboard
On demand, MVP flow:
1. Read `evidence.jsonl`, dedup by structured key, **validate each against current
   repo** (dep still in manifest? file still exists?). Stale drops out. Build
   `confirmed` items.
2. Run a **light LLM inference pass** over touched files → `inferred` / `possible`
   concept items with `parent` + file:line evidence. (Where "React" becomes
   "useEffect", "JSX", "props" — the granularity that matters.)
3. Write `inventory.json` = **everything** (stable order, only if changed).
4. Start local server, print: `Open http://localhost:3874`.

`/ontrack` regenerates data only — it does NOT generate HTML, and never asks
knowledge questions in chat. The dashboard owns display + marking.

### 3. Dashboard — local server + `dashboard.html` (committed template)
`dashboard.html` is a **static, data-driven template committed to the repo** (it's
source code, not a generated artifact — churns only when the UI improves). At
runtime it `fetch`es `inventory.json` (project truth) + `personal.json` (user
status) from the server and renders client-side. Sections: **To Review / Learning /
Learned / Ignored**, nested by `parent`. Each item card: name · what · used-in ·
search query · status radios.

Radios per item: `[ ] Known  [ ] Somewhat  [ ] To learn  [ ] Ignore`.
Click → JS updates section live (optimistic) + `POST`s to persist.

Server scope (local-only + security):
- **Stdlib only** (Python `http.server` / Node one-file). No deps. Serves the static
  `dashboard.html` + `inventory.json` + `personal.json` (GET) and one POST. JSON
  validation + port fallback + atomic write + path safety is real work — small but
  not trivial; don't underestimate.
- **Bind `127.0.0.1`, port `3874`.** Never `0.0.0.0` — file-writing endpoint stays
  off the network. Port in use → increment and reprint.
- **One route: `POST /status`** → `{id, status}` → validates status against the
  4 allowed values, writes `personal.json` **atomically** (temp + rename). Only file
  it ever writes. `inventory.json` read-only here.
- Live DOM re-sorts on click; no page reload, no `/ontrack` re-run to save answers.
- Lifecycle: dies on idle timeout / Ctrl-C. Add a `/done` route later if users want
  it kept open.

**Boundary held:** dashboard shows what exists + what to search. Does NOT teach.

The loop: build → `/ontrack` → open dashboard → mark status → sections re-sort →
go learn the "Learning" pile elsewhere.

## On "does the user know it?" — the tool does not guess (v1)
AI cannot read the user's mind; same code can mean expert or clueless. So v1 makes
no claim. Dashboard shows everything used; the **user** self-classifies via status
radios → `personal.json`. Author-signal ("Claude wrote this untouched — you may not
know it") is a *later* hint, never a claim. See deferred "coach" below.

## Deferred (NOT v1 — over-build for the stated goal)
- **Roadmap generator**: order unknowns into a learning sequence. Add if a flat
  list proves insufficient.
- **Used-vs-learned coach**: estimate engagement from author signals (hand-edited
  vs Claude-wrote-untouched). Weak proxies only — defer until core proves useful.

## Packaging
Shipped as a Claude Code **plugin** in the `ontrack/` subdir; this repo is also the
**marketplace** (`.claude-plugin/marketplace.json`). Install:
`/plugin marketplace add NoName7n11/OnTrack` → `/plugin install ontrack@ontrack`.
Plugin-internal command paths use `${CLAUDE_PLUGIN_ROOT}`; data still writes to the
**user project's** `.ontrack/` (hooks/skills run with the project as cwd).

Codex uses the same implementation through `codex/ontrack.py` plus `AGENTS.md`.
This is intentionally local and explicit: Codex runs `snapshot`, writes
`.ontrack/concepts.json`, runs `build`, and starts `serve` when the user wants the
dashboard.

## Build order
1. Data format: `evidence.jsonl` line schema + `inventory.json` schema. Format first.
2. Tracker hook → writes confirmed evidence. Prove passive capture.
3. `/ontrack` builds `inventory.json` from confirmed evidence. Prove core loop.
4. Dashboard + server: render sections, radios, `POST /status` → `personal.json`.
5. Add LLM inference pass → `inferred`/`possible` concepts + `parent`. The value. **[done]**
6. (Only if wanted) roadmap ordering; used-vs-learned coach.

## Verification
- Tracker: build a throwaway React+auth project in a session → confirm
  `evidence.jsonl` auto-fills (react, jsonwebtoken, tsx) with zero manual input.
  (Evidence lines carry no `confidence`; that field lives on inventory items.)
- Inventory: run `/ontrack` → `inventory.json` lists `confirmed` inventory items
  for libs + `inferred` concepts (with `id` + `parent`), each with what/where/search.
- Dashboard: open `localhost:3874` → items grouped To Review; click "Known" on
  React → moves to Learned live, `personal.json` gets `"library:react":"known"`, and
  `concept:react/useeffect` stays in To Review (no cascade).
- Isolation: confirm the server only ever writes `personal.json`; `inventory.json`
  byte-identical after marking.
- Honesty: no `inferred` item ever appears in `evidence.jsonl` (hook writes facts).
- Staleness: remove a dep from `package.json` → next `/ontrack` drops it from
  inventory though its `evidence.jsonl` line remains.
